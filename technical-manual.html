Request Flow

  1. React Native app makes HTTP request to DRF API or opens WebSocket connection.
  2. DRF views authenticate via JWT, process business logic, write to PostgreSQL.
  3. For real-time events, views publish to Redis channel layer via Django Channels.
  4. WebSocket consumers push updates to connected clients instantly.
  5. Celery tasks run async operations (dispatch, notifications, auto-cancels) independently.
  6. Monnify webhooks hit a dedicated endpoint for payment confirmation.

Order Lifecycle

Status Flow

draft → pending → preparing → ready → delivery → arrival → completed

Pickup orders follow a slightly different path: pending → preparing → pickup_ready → completed

Rejected orders: pending → rejected (triggers immediate refund)

Status Transition Rules

StatusSet BySide Effects
draftSystem (order creation)No side effects; order not yet paid
pendingMonnify webhook / verify_paymentVendor push notification, vendor WS event, streak awarded, FoodysRevenue created
preparingVendor (accept action)Prep time timer starts, preparation_started_at set, estimated_prep_time calculated in seconds
readyVendor (mark ready)Vendor payout #1 triggered, batch created/joined, dispatch engine queued, actual_prep_time recorded
pickup_readyVendor (pickup flow)Vendor payout #2 triggered, customer push with confirmation code
deliveryRider (confirms pickup)picked_up_at set, customer notified
arrivalRider (marks arrived)arrived_at set, customer push, 10-minute auto-complete countdown starts
completedRider (code confirmation) or auto-completeRider payout triggered, delivery_time_minutes calculated, customer prompted to review
rejectedVendor (reject) or system (auto-cancel at T+10)OrderRefund created, process_order_refund task fired, customer WS event, push notification

Pending Order Auto-Cancel

The check_pending_orders task runs every 30 seconds and escalates unacknowledged paid orders:

Time Since PaymentActionFlag Set
T + 3 minSoft reminder push + WebSocket to vendorpending_reminder_1_sent = True
T + 6 minUrgent reminder push + WebSocket to vendorpending_reminder_2_sent = True
T + 10 minAuto-cancel: order → rejected, refund initiated, both parties notified, auto_cancelled = True—
ℹ️ Age is calculated from payment_verified_at, not created_at. Orders are created before payment completes, so the timer only starts once Monnify confirms the payment.

Rider Assignment & Dispatch

Batching Strategy

Orders from the same restaurant are grouped into a batch before dispatch to optimise rider routes and reduce per-delivery cost.

ConstantValueMeaning
BATCH_WINDOW_SECONDS180 (3 min)How long to collect orders before dispatching
BATCH_MAX_ORDERS5Force-dispatch when batch reaches this size
MAX_ORDER_WAIT_SECONDS420 (7 min)Max time a ready order can wait before urgent dispatch

Rider Selection Algorithm

The dispatch engine runs a two-pass selection on every call to run_dispatch(campus):

Pass 1 — Zone match
  Filter riders in the correct zone group for this restaurant
  Pick the rider with fewest active orders

Pass 2 — Campus fallback (if Pass 1 yields nothing)
  Consider all available riders on campus
  Pick the rider with fewest active orders

If no rider found:
  Queue order in OrderAssignmentQueue
  Retry every 60 seconds or immediately when any order completes

Zone Groups (Veritas University)

GroupZonesCoverage
ABA, B, CBoys Cafe area, boys hostels (A–Q), Ase, MunchBox
DED, E, F, G, H, ICampus perimeter, Girls Cafe, Law, Clinic, Kelson, Health Sciences, New buildings

Delivery Ordering (Sequence)

Once a rider is assigned, orders within a batch are sorted by delivery fee (lowest = nearest) to give the rider an optimised route. This sequence is stored in Order.delivery_sequence and drives the display order in the rider app.

Dispatch Triggers

The dispatch_trips Celery task runs every 60 seconds per campus and handles three scenarios in priority order:

  1. Mature batches — batches whose collection window has expired (≥ 3 minutes old)
  2. Urgent unbatched orders — ready orders waiting longer than MAX_ORDER_WAIT_SECONDS without a rider
  3. Queued retries — orders in OrderAssignmentQueue from previous failed assignments
✅ Retry acceleration: retry_queued_orders() is also called on_commit whenever any order completes or is rejected, giving queued orders an immediate retry without waiting for the 60-second cron cycle.

Redis as Channel Layer

Django Channels uses Redis as its channel layer backend. Redis is also used by Celery as its message broker and result backend. Both share the same Redis instance on Render.

⚠️ If Redis goes down, both real-time WebSocket messaging and Celery task processing will fail simultaneously. Monitor Redis health as a critical dependency.

Payment Integration (Monnify)

Transaction Flow

  1. Checkout screen calls POST /payments/create-checkout/ with orderId.
  2. Backend initialises transaction with Monnify, then creates a Virtual Account for bank transfer.
  3. Frontend displays account number, bank name, amount, and expiry to customer.
  4. Customer transfers the exact amount.
  5. Monnify fires a SUCCESSFUL_TRANSACTION webhook to /payments/webhook/monnify/.
  6. Backend verifies signature, then verifies transaction with Monnify API.
  7. On success: Payment marked paid, order moved to pending, vendor notified.
  8. As a fallback, POST /payments/verify/ is available for manual polling from the frontend.

Refund Flow

Order rejected / auto-cancelled
    → OrderRefund record created (status: pending)
    → process_order_refund.delay(order_id) fired
    → Task calls Monnify /api/v1/refunds/initiate-refund
    → On success: refund.status = 'processing'
    → Monnify sends funds back to customer's bank account
    → Max retries: 3, with 60s delay between attempts
    → Permanent errors (permission denied) do not retry
🔒 Fixie proxy: All outbound requests to Monnify (refunds, disbursements, token fetch) are routed through Fixie to present a fixed outgoing IP address. Monnify whitelists this IP. Only the "https" key is needed in the proxies dict. FIXIE_URL format: http://user:password@host:port.

Vendor Payouts (Batch Disbursement)

Vendor payouts use Monnify's batch disbursement API (/api/v2/disbursements/batch). Three manual windows per day plus automatic nightly processing at 10:30 PM. Rider payouts follow the same mechanism.

Monnify Refund Reason Length

⚠️ Monnify enforces a 64-character maximum on refundReason. Always trim to [:64] before sending. Using [:100] will cause a 400 Refund Reason must be between 1 and 64 characters error.

Restaurant

FieldPurpose
statusOPEN / CLOSED / OFFLINE
offline_untilDateTimeField — when the restaurant auto-resumes from a pause
auto_openBoolean — whether the restaurant auto-opens at opening_time each day
pickup_enabledBoolean — whether pickup fulfilment is available

OrderBatch

Groups orders from the same restaurant for a single rider trip. States: pending → dispatched → completed. Only delivery orders are batched; pickup orders are never added.

OrderAssignmentQueue

Holds orders that failed rider assignment. Tracks retry_count and last_tried_at. Cleared when a rider is successfully assigned.

OrderStreak

One per user. Tracks current_streak, longest_streak, last_order_date, broken streak metadata, and free restore counters. Streak increments when order moves to pending (payment confirmed).

⚠️ Single worker process: --beat runs the Beat scheduler inside the same worker process. This is acceptable at current scale. Do not run separate celery beat and celery worker processes simultaneously — it will cause duplicate task execution.

Post-Deploy Checklist

  1. Run python manage.py migrate
  2. Run python manage.py seed_periodic_tasks
  3. Restart the background worker (to pick up new model fields)
  4. Verify Celery Beat tasks are registered in Django admin → Periodic Tasks
  5. Test Monnify webhook endpoint is reachable
  6. Verify Fixie proxy is routing correctly for Monnify refund calls

Local Development

# Backend
python manage.py runserver

# Celery (local — solo pool avoids duplicate delivery sequences)
celery -A foodysbackend worker --beat --loglevel=info --pool=solo

# PostgreSQL
brew services start postgresql  # Homebrew managed
🚨 Always use --pool=solo locally. Running without it causes duplicate delivery_sequence values in batch dispatch due to concurrent worker threads.

Development Guide & Known Gotchas

Known Gotchas

🚨
Stale Celery worker after migrations
If you add a new field to a model and run migrations, the running Celery worker still has the old model definition in memory. Always restart the worker after any schema change or you'll get silent failures or FieldError exceptions in tasks.
🚨
Duplicate dispatch notifications
The dispatch query must include both rider__isnull=True AND trip_notified=False filters. Missing either causes duplicate rider notifications. This was a historical bug — both filters are required.
⚠️
MultipleObjectsReturned in test scripts
When querying users by username in management commands or test scripts, use email or ID instead. Usernames are not guaranteed unique at the DB level — querying by username can silently return the wrong user or raise MultipleObjectsReturned.
⚠️
Shield-aware streak reset
The reset_broken_streaks task uses a 2-day cutoff from today (not 1-day) to avoid zeroing streaks that are still within the 24-hour restore window. If you change the restore window, update the cutoff accordingly.
ℹ️
Streak increment on pending (not completed)
Streaks are incremented when an order moves to pending (payment confirmed), not completed. This is an intentional product decision — the customer has committed by paying. Do not change this without product review.

Testing Async Tasks

Use management commands or shell scripts to verify task behaviour rather than waiting for Celery Beat. For example, to test the pending order auto-cancel:

# Create a test order via the app, pay, then call the task directly:
from orders.tasks import check_pending_orders
check_pending_orders.apply()

# To reset test orders between runs:
python manage.py reset_orders

Adding a New Periodic Task

  1. Write the task in the relevant tasks.py file decorated with @shared_task.
  2. Add an entry to the tasks list in seed_periodic_tasks.py.
  3. Run python manage.py seed_periodic_tasks.
  4. Restart the Celery worker.
  5. Verify in Django admin → Periodic Tasks that the new task appears and is enabled.
